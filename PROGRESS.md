# 한끗루틴 진행 상황

> 새 대화(세션)를 시작할 때 이 파일을 먼저 읽으면 지금까지 뭘 했고 뭐가 남았는지 바로 파악할 수 있어요.
> 실제 코드/DB 상태는 git 커밋과 Supabase가 정답입니다 — 이 문서는 "요약"일 뿐이니, 내용이 코드와 다르면 코드를 신뢰하세요.

## 프로젝트 구조

- `index.html` — 메인 앱 (청소년/끗짱 공용 SPA, GitHub Pages 배포)
- `admin.html` — 관리자 전용 웹페이지 (참가자 관리, 루틴 관리). 로그인은 앱과 동일한 Supabase Auth 사용, `ADMINS` 배열로 접근 제한
- `landing.html` — 마케팅 랜딩페이지
- `supabase-schema.sql` — DB 스키마 + **모든 마이그레이션 기록** (파일 맨 아래로 갈수록 최신). 새 변경사항은 항상 이 파일 끝에 `[마이그레이션 YYYY-MM-DD]` 블록으로 추가하고, Supabase SQL Editor에서 그 블록만 실행하는 방식으로 운영 중
- 배포: GitHub Pages, repo `youthitdev/youthit-routine`, 브랜치 `main`
- DB: Supabase (`ynqvhsffoesjzefitafv` 프로젝트)

## 역할 & 관리자

- 역할: `youth`(청소년), `kkutjjang`(끗짱)
- **관리자 이메일 (dev@youthvoice.or.kr, yv@youthvoice.or.kr)** — `index.html`의 `ADMINS` 배열, `admin.html`의 `ADMINS` 배열, DB `is_admin()` 함수 **3곳 모두 동일하게 유지해야 함**. 하나라도 다르면 "화면엔 관리자로 보이는데 실제 DB 쓰기는 실패"하는 버그가 생김 (실제로 한 번 발생했었음)
- 관리자는 `state.loginType='both'`로 청소년/끗짱 화면을 전환하며 이용 가능

## 완료된 주요 기능

### 가입 · 인증(계정) 체계
- 가입 시 기본정보(닉네임/실명/생년월일/휴대폰/지역 시도·시군구/역할) 수집, 청소년은 재학상태(학교밖/재학/대안학교)까지
- **끗짱 가입 승인제** — 가입 즉시 이용 불가, 관리자 승인(`kkutjjang_status`) 후 이용 가능. 관리자 이메일은 자동 승인
- 학교밖청소년 서류 제출 → 관리자 승인 플로우 (Storage 비공개 버킷)
- 재학상태별로 요구 서류 안내 차등화 (학교밖: 정원외관리증명서 등 5종 / 재학: 청소년증·학생증)
- 승인되면 이름 옆 "✅ 학교밖청소년 인증" 뱃지 노출
- **비밀번호 찾기/재설정** 플로우 (Supabase resetPasswordForEmail + 재설정 화면)
- 프로필 편집(닉네임/실명/생년월일/휴대폰/지역, 청소년은 재학상태도) — Supabase에 실제 반영

### 루틴
- 루틴 생성/수정 (끗짱), 참여자격 분리("청소년 누구나" / "학교밖청소년 전용")
- **루틴 개설은 관리자만** 가능(당분간), 관리자가 담당 끗짱(`led_by`)을 지정
- 담당 끗짱은 자기 루틴만 보고 관리(홈/루틴탭/인증탭/통계 전부 스코핑됨), 관리자는 전체 조회
- 참여 신청 시 각오/소감 입력, 끗짱이 승인/거절
- **참여 승인 안 된 사람은 인증 불가** (클라이언트+RLS 이중 방어 — 한 번 뚫려있던 취약점 수정함)
- OT(첫날)/클로징(마지막날) 온라인 모임 일정 + 줌 링크, 구글캘린더 저장 연동
- 루틴 상태(`recruit→active→done`) 중 `active→done`은 **종료일 기준 자동 전환** (클라이언트 계산 + 권한 있는 사용자가 조회 시 DB에도 반영)
- 파트너 기관명, 루틴 대표 이미지, 카드 UI(이미지 중심 디자인)
- **루틴 보관/삭제** — admin.html에서 보관(숨김) 또는 완전 삭제 가능

### 인증(루틴 인증) · 커뮤니티
- 인증 사진(최대 5장, Storage 업로드)·소감 실제 저장 (예전엔 저장 안 되던 버그 있었음)
- 인증 피드가 **실데이터 연결됨** (예전엔 `CERT_FEED`라는 하드코딩 목업이었음) — 좋아요/댓글 실시간 반영
- 끗짱도 자기 루틴을 청소년처럼 직접 인증 가능
- 게시글/댓글 삭제 — 본인 또는 관리자만 (RLS + UI)

### 관리자 페이지 (admin.html)
- 청소년/끗짱 목록 검색·필터(지역/재학상태/서류상태/참여루틴)
- 일괄 승인/반려 (청소년 서류, 끗짱 가입)
- CSV 내보내기
- 루틴 탭: 상태 필터, 보관/삭제
- **통계 대시보드** (`📊 통계` 탭, Chart.js 사용) — KPI(청소년/끗짱 수, 진행중 루틴, 전체 완주율), 지역별 청소년 분포/재학상태 비율/루틴 상태 분포 차트, 루틴별 완주율 테이블. 완주율은 각 루틴의 실제 기간(`end_date - start_date`) 기준 2/3 이상 인증 여부로 계산 (기존 index.html의 21일 하드코딩 로직을 일반화함)

### 기타
- 모바일 하단 탭이 안 고정되던 버그 수정 (`.app`의 `transform: translateZ(0)`가 원인)
- 랜딩페이지 브랜딩(YouthVoice 표기, 로고, 파비콘)
- 홈 화면 "이번 주 한끗" 전역 섹션 제거 → "참여 중인 루틴" 각 카드에 루틴별 주간 인증 그리드로 통합 (2026-07-10, 스트릭 배지·진행률과 중복돼 보여서 정리, 루틴이 보통 1~2개뿐이라 루틴별로 나누는 게 더 명확하다는 논의 끝에 결정)

## 남은 작업 (우선순위순)

1. **알림 인프라 (Web Push) — ✅ 1·2·3단계 완료 (2026-07-10, 관리자 알림 보내기까지 실기기 검증됨)**
   - 로드맵: ~~①구독+테스트 발송 파이프라인~~ → ~~②트리거 연결(참여 승인/거절, 좋아요, 댓글)~~ → ~~③관리자 수동 공지 발송~~ → **④마일스톤(5일차/10일차)·리마인드 시간 설정(pg_cron 스케줄러) ← 다음**
   - 구축된 것: `push_subscriptions` 테이블+RLS, `send-push` Edge Function(관리자 JWT 또는 service 키만 발송, 만료 구독 자동 정리, `user_id`/`user_ids`/`target:'all'`/`target:'routine'+routine_id` 지원, CORS 헤더 포함), VAPID 시크릿, 마이페이지 "알림 설정" 토글, service-worker.js push/notificationclick 핸들러(캐시 v4), **DB 트리거 3종**(참여 승인/거절→`notify_push()`, 인증 좋아요/댓글→작성자 알림, pg_net으로 비동기 호출), **admin.html "🔔 알림 보내기" 탭**(전체/루틴별 대상 선택 + 제목·내용 입력 발송, `sb.functions.invoke` 사용)
   - 발송 방법(재사용): `POST {SB_URL}/functions/v1/send-push` + `Authorization: Bearer <service키 또는 관리자 JWT>` + `{"user_id"|"user_ids"|"target":"all"|{"target":"routine","routine_id":N}, "title", "body", "url"}`. DB에서 직접 보낼 땐 `SELECT notify_push(user_id, title, body);`
   - 디버깅용: 발송 시도 로그는 `SELECT * FROM net._http_response ORDER BY id DESC LIMIT 5;`. admin.html에서 브라우저로 함수 호출 시 CORS 헤더 필수(한 번 빠뜨려서 "Failed to send a request" 에러 났었음 — 이제 고쳐짐)
   - 로컬 도구: Deno(`~/.deno/bin`)·Supabase CLI(`~/.local/bin`, login+link 완료) — PATH 미등록, 새 터미널 열 때마다 `export PATH="$HOME/.deno/bin:$HOME/.local/bin:$PATH"` 필요. VAPID 키 원본 `~/.hankkut-vapid-keys`(**비밀키 절대 저장소에 넣지 말 것**)
   - 주의사항: SQL Editor 실행 시 **프로젝트가 ynqvhsffoesjzefitafv인지 꼭 확인**(다른 프로젝트 approval에 실행한 사고 있었음 — 그쪽 push_subscriptions 테이블 DROP 필요). iOS는 "홈 화면에 추가"해야만 푸시 동작(16.4+), 집중 모드 중엔 배너 없이 알림 센터에만 쌓임. 새 테이블 만들면 API 스키마 캐시 갱신에 `NOTIFY pgrst, 'reload schema';` 필요할 수 있음
